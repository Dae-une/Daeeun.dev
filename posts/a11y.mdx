# 웹 표준과 웹 접근성

웹 접근성은 모든 사용자가 신체적,환경적 조건에 관계없이 웹에 접근하여 이용할 수 있도록 보장하는 것이다.
즉 누구나 기능을 사용할 수 있어야 한다.<br/>

주로 접근성에 대한 논의를 신체적 장애가 있는 사용자에게 초점을 맞추는 경향이 있지만,<br/>
우리 모두는 어떠한 이유든 액세스할 수 없는 인터페이스를 접할 수 있다.<br/>
특정 지역,네트워크 환경, 접근 장치에 따라 제대로 동작하지 않는 경우를 볼 수 있으며, 모두 접근성 문제이다.<br/>

웹 표준은 웹 접근성 수준을 향상시키기 위한 다양한 방법론 중 가장 최적의 방법론으로 국내 웹 관련 업계에서 관심을 모으고 있다.<br/>
흔히 생각하는 표준과는 다르게 웹에서의 표준은 W3C의 토론을 통해 나온 권고안을 말한다<br/>

# 접근성 지침

[WCAG(Web Content Accessibillity GuidLines)](https://www.w3.org/TR/WCAG20/)를 요약하면 아래와 같다.

## 인지성 (Perceivable)

정보와 사용자 인터페이스 요소는 그들이 인지할 수 있도록 사용자에게 표시될 수 있어야 한다.<br/>
이는 시각과 같은 특정 감각으로 무언가를 인지할 수 있다고 해서 모든 사용자가 인지할 수 있다는 의미는 아니다.<br/>

- 텍스트가 아닌 콘텐츠에 적절한 대체 텍스트를 제공해야한다.
- 동영상과 같이 시각을 바탕으로 한 미디어에 대한 대안을 제공해야한다.
- 정보와 구조의 손실 없이 콘텐츠를 다른 방식들로 표현할 수 있어야 한다.
- 사용자들이 보다 쉽게 보고 들을 수 있는 전경에서 배경을 분리한 콘텐츠를 만들어야 한다.

## 운용성 (Operable)

사용자 인터페이스 요소와 탐색은 운용 가능해야 한다.<br/>

- 키보드로 모든 기능을 사용할 수 있도록 해야 한다.
- 읽기 및 콘텐츠를 사용하는 사용자에게 충분한 시간을 제공해야 한다.
- 사용자가 탐색하고, 컨텐츠를 찾고, 그들이 어디에 위치하고 있는지를 알 수 있도록 도와주는 방법을 제공해야 한다.

## 이해성 (Understandable)

정보와 사용자 인터페이이스 운용은 이해할 수 있어야 한다.<br/>
사용자가 인터페이스를 이해하는데 어려움이 없으며, 일관성을 가지고 있도록 해야한다.<br/>

- 텍스트 컨텐츠를 읽고 이해할 수 있도록 해야 한다.
- 웹페이지를 예측 가능한 방식으로 표시하고 작동하게 해야한다.
- 사용자의 실수를 방지하고 수정할 수 있도록 해야한다.

## 견고성 (Robust)

다양한 환경,브라우저,장치 등에서 컨텐츠를 사용할 수 있어야 한다.<br/>

# 접근성이 좋은 사이트 만들기

## 시맨틱 태그

모든 레이아웃을 `div`태그로 만드는 대신 태그 자체에 의미를 부여해서 알맞은 위치와 용도로 사용하도록 한다.<br/>
대표 적으로 `nav`,`footer`,`section`등이 있다.

시맨틱 태그를 사용하면

- 프로젝트를 수정하기 위해 모든 태그를 검색할 필요 없이 원하는 태그를 빠르게 찾을 수 있다.
- 검색엔진은 비의미적 태그보다 시맨틱 태그를 더 중요하게 여겨 SEO에 더 좋다.

## 포커스

포커스는 개발자가 의도적으로 막지 않는 이상 기본적으로 제공되는 기능이다.<br/>
링크,버튼,인풋 등 모든 폼 컨포넌트에는 포커스에 대한 명확한 상태가 필요하다.<br/>
사람들은 이런 정보들에 의존해서 데이터를 입력하고 상태를 확인한다.

## 버튼

버튼을 `div`태그를 이용하여 만들지 않는다.<br/>
`div`태그는 포커스를 맞출 수 없으며 키 이벤트가 활성화 되지 않는다.<br/>
만약 `div`태그로 만들어야 한다면 `tabindex`를 통해서 탭할 수 있게 만들어 주자.<br/>

택스트가 없는 버튼에는 `aria-label`을 추가해주자.<br/>
버튼에 텍스트가 없으면 `aria-label`을 이용하여 버튼의 동작을 간략하게 설명한다.

## 대체 텍스트

`img`태그를 사용할때, 이미지가 시각적으로 전달하는 내용을 `alt`를 이용하여 표시하자.<br/>
만약 단순한 백그라운드 이미지라면 `background-image`로 이미지를 제공하자.<br/>
또한, 이미지에 마우스를 올리면 `title`속성이 나타난다.

## 링크

링크에 대한 상태관리가 되어야 한다.<br/>
방문했을때, 하지 않았을때 서로 적절한 스타일을 적용해야 한다.<br/>
또한 사용자가 링크와 상호작용할 때도 중요하다.<br/>
포인트 커서 나 포커스를 주도록 하자.

## WAI-ARIA

> Web Accessibility Initiative – Accessible Rich Internet Applications<br/>
> WAI는 W3C에서 웹 접근성을 담당하는 기관으로 ARIA는 RIA환경의 웹 접근성에 대한 표준 기술 규격을 의미

RIA는 정적인 HTML,단순한 자바스크립트 환경의 웹이 아닌 동적인 자바스크립트와 Ajax와 같은 기술을 <br/>
사용한 환경에서 수준 높은 UX를 제공하는 웹 애플리케이션 이다.

WAI-ARIA는 스크린리더가 브라우저를 읽을 때 각 요소가 어떤 역할을 하는지, 무슨 의미로 존재하는지 <br/>
알 수 있도록 하기위해 만들어진 기술이다.<br/>
또한, 자바스크립트나 다른 동적 언어로 인해 페이지의 요소가 바뀌더라도 새로고침을 하지 않아도 스크린 리더가 바뀐 요소를 읽어준다.

### role

`role`은 태그의 역할을 알려주는 속성이다.<br/>
`role`에 들어가는 값은 [MDN](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/ARIA_Techniques)을 통해 확인 가능하다.<br/>

### aria-label

`aira-label`은 태그가 가지고 있는 의미를 적어주는 이름표 같은 역할을 한다.<br/>

`aria-label`은 `role`의 banner 나 tap 처럼 값이 정해져 있는게 아닌, 브라우저가 스크린 리더 사용자에게 <br/>
전달해야할 내용을 상황에 따라 적어야한다.

예를 들어, collapse 아이콘이 있다고 가정하자.
우리는 화면에 보이는 아이콘으로, 또는 아이콘의 위치나 스타일로 해당 버튼이 무슨 역할을 하는지 <br/>
짐작할 수 있지만, 스크린 리더 사용자는 아이콘의 생김새도, 의미도 알기 어렵다.<br/>

하지만 만약 `aria-label='펼치기'`를 입력해준다면, 스크린리더는 그것을 읽어 정보를 전달해준다.<br/>
여기에 `role`을 추가해주면

```jsx
<div>
  <a href="#" aria-label="펼치기" role="button">
    <i class="ico ico-arrow"></i>
  </a>
</div>
```

버튼 이라는것도 전달해주고, 펼치기 기능을 한다는 정보까지 전달해 줄 수 있다.<br/>

`aira-label`을 사용할 때에는, 시맨틱 태그 이거나,시맨틱 태그가 아닌경우 role 속성 값이 interactive 하는 값일때 사용할 수 있다.

더 자세한 정보는 [MDN](https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/Attributes/aria-required)을 참고하길 바란다.
